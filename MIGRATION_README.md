# 数据库迁移说明

## 概述

本迁移脚本用于将旧的数据库结构迁移到新的统一表结构。

### 旧结构 → 新结构映射

| 旧表结构 | 新表结构 | 说明 |
|----------|----------|------|
| `model_name_mapping` | `models` | 模型元数据表 |
| `{model}_requests` | `model_requests` | 所有模型的请求记录合并到一个表 |
| `{model}_runtime` | `model_runtime` | 所有模型的运行时间合并到一个表 |
| `{model}_tier_pricing` | `tier_pricing` | 所有模型的阶梯计费合并到一个表 |
| `{model}_hourly_price` | `hourly_pricing` | 所有模型的按时价格合并到一个表 |
| `{model}_billing_method` | `billing_methods` | 所有模型的计费方式合并到一个表 |
| `program_runtime` | `program_runtime` | 保持不变 |

### 迁移内容

- ✅ 程序运行时间记录
- ✅ 模型映射关系
- ✅ 所有请求记录（包含 token 统计）
- ✅ 模型运行时间记录
- ✅ 阶梯计费配置
- ✅ 按时计费配置
- ✅ 计费方式设置

## 使用方法

### 1. 预览迁移计划（推荐先执行）

在正式迁移前，先查看有哪些数据会被迁移：

```bash
python migrate_database.py --dry-run
```

输出示例：
```

开始数据库迁移
旧数据库: webui/monitoring.db
新数据库: webui/monitoring.db
迁移模式: 演习模式（不执行）
============================================================
发现 12 个模型需要迁移
演习模式：迁移计划预览
- 模型数量: 12
  - Qwen3.5-27B
  - Qwen3-4B-Instruct-2507
  ...
演习模式结束，未执行实际迁移
```

### 2. 执行迁移（自动备份）

**推荐方式**：自动备份旧数据库并原地迁移：

```bash
python migrate_database.py --backup
```

这会：
1. 自动创建备份文件 `monitoring.db.backup.{时间戳}`
2. 执行迁移
3. 用新数据库替换旧数据库
4. 保留旧数据作为备份

### 3. 迁移到新文件（更安全）

如果你想保留旧数据库不变，创建一个新数据库：

```bash
python migrate_database.py --output webui/monitoring_new.db
```

然后测试新数据库：
```bash
# 启动程序时指定新数据库
# 或手动替换旧数据库
```

## 命令行参数

```
--input PATH      旧数据库路径（默认: webui/monitoring.db）
--output PATH     新数据库路径（默认: 覆盖旧数据库）
--backup          在迁移前自动备份旧数据库
--dry-run         演习模式：只显示迁移计划，不执行实际迁移
```

## 迁移后的验证

迁移完成后，建议进行以下验证：

### 1. 启动程序测试

```bash
python main.py
```

检查日志是否正常，数据库是否正确加载。

### 2. 检查数据完整性

可以临时使用 SQLite 工具查看新数据库：

```bash
# 检查表是否创建成功
sqlite3 webui/monitoring.db ".tables"

# 应该看到: models, model_requests, model_runtime, tier_pricing,
#           hourly_pricing, billing_methods, program_runtime

# 检查数据是否迁移成功
sqlite3 webui/monitoring.db "SELECT COUNT(*) FROM model_requests"
sqlite3 webui/monitoring.db "SELECT original_name FROM models"
```

### 3. Web UI 功能测试

- 查看模型列表
- 查看请求数据统计
- 查看计费配置
- 测试新请求是否正确记录

## 回滚方法

如果迁移后发现问题，可以使用备份文件回滚：

```bash
# 停止程序

# 恢复备份
cp webui/monitoring.db.backup webui/monitoring.db

# 或使用时间戳备份
cp webui/monitoring.db.backup.1234567890 webui/monitoring.db

# 重启程序
```

## 注意事项

1. **备份优先**：迁移前务必备份，或使用 `--backup` 参数
2. **停止程序**：迁移时请确保 LLM-Manager 已停止运行
3. **磁盘空间**：确保有足够磁盘空间（约为原数据库大小的 2-3 倍）
4. **时间估算**：迁移时间取决于数据量，通常几秒到几分钟

## 常见问题

### Q: 迁移会丢失数据吗？
A: 不会。迁移脚本会完整复制所有数据到新表结构。

### Q: 迁移失败怎么办？
A: 如果使用了 `--backup`，备份文件会保留。如果失败，数据库不会被修改。

### Q: 可以多次运行迁移脚本吗？
A: 不建议。新结构的数据库不能再次迁移。如需重新迁移，请先恢复备份。

### Q: 旧数据库文件会被删除吗？
A: 不会。旧数据库会被重命名为 `.backup`（如果使用 `--backup`），或保持不变（如果使用 `--output`）。

### Q: 迁移后程序启动报错？
A: 检查是否使用了新的 `data_manager.py`。确保代码和数据库版本匹配。
